# 囚徒健身进度追踪网页 - 完整设计方案

## 一、Root（产品目标）

基于《囚徒健身》六艺十式体系，打造一款**纯前端、本地优先**的进度追踪网页应用，帮助用户：
- 清晰追踪六艺各式的完成情况与进阶路径
- 每日打卡、记录组数/次数，自动判断阶段完成
- 通过统计与成就获得正反馈
- 生成可分享的进度卡片（当前阶段、耗时、最终技完成情况）

---

## 二、节点1：六艺进度系统设计

### 2.1 六艺分类

| 艺 | 英文/代号 | 主要肌群 |
|----|-----------|----------|
| 俯卧撑 | Pushup | 胸、肱三头肌、三角肌前束 |
| 深蹲 | Squat | 股四头肌、臀、腘绳肌 |
| 引体向上 | Pullup | 背阔肌、肱二头肌 |
| 举腿 | LegRaise | 腹直肌、核心 |
| 桥 | Bridge | 脊柱、下背 |
| 倒立撑 | HandstandPushup | 肩、上肢 |

### 2.2 每艺的阶段划分（十式）

每艺分为 **10 式**，对应 10 个递进难度等级：
- **第 1～3 式**：初级（入门）
- **第 4～6 式**：中级（进阶）
- **第 7～9 式**：高级（精进）
- **第 10 式**：最终技（Master）

完成标准采用书中「升级标准」：例如某式需完成指定组数×次数（如 3 组×若干次），且能**连续 2 次训练**达标，则允许升级到下一式。

### 2.3 阶段完成标准（可配置）

- **单式完成**：用户在某式上勾选「已达标」或系统根据打卡记录自动判定（如最近 2 次打卡均达到该式目标）。
- **阶段完成**：完成该艺下该「段」内所有式（如初级 = 1～3 式全部达标）。
- **最终技完成**：该艺第 10 式达标，触发「最终技」成就与庆祝。

### 2.4 最终技定义与庆祝机制

- **最终技**：每艺的第十式（单臂俯卧撑、单腿深蹲、单臂引体向上等）。
- **庆祝机制**：弹窗/全屏庆祝动画、成就徽章、解锁「分享卡片」中「已完成最终技」标签。

## 三、节点2：数据记录系统

### 3.1 本地数据存储方案

- **技术选型**：`localStorage` + 结构化 JSON；可选后续加 `IndexedDB` 应对大量历史记录。
- **存储键**：
  - `cc_progress`：用户进度（每艺当前式、每式达标日期、历史达标记录）。
  - `cc_checkins`：每日打卡记录（日期、每艺的式、组数、次数、是否达标）。
  - `cc_settings`：用户设置（提醒、单位、引导完成标记等）。
  - `cc_achievements`：成就解锁与解锁时间。

### 3.2 每日打卡功能

- 支持选择日期（默认今天），按「艺」+「式」记录：
  - 式选择、组数、每组次数、是否自评达标。
- 一次打卡可包含多艺多条记录（同一天多艺多式）。
- 未达标也可记录，用于趋势与「连续打卡」统计。

### 3.3 进度追踪算法

- **当前式**：每艺维护一个「当前挑战式」（用户手动设置或由系统根据「最近达标」推断）。
- **达标判定**：
  - 若用户勾选「本次达标」且当前式与选择的式一致，则记录该式达标日期。
  - 可选：若连续 2 次打卡该式且都勾选达标，则自动标记该式完成并建议升级。
- **升级**：用户可手动「升级到下一式」，或由系统在满足条件时提示升级。

### 3.4 阶段完成时间计算

- 每式存储「首次达标日期」。
- 阶段耗时 = 该阶段最后一式的达标日期 − 该阶段第一式的达标日期。
- 若某式未达标则该项不参与统计，或显示「进行中」。

## 四、节点3：用户界面设计

### 4.1 首页

- **当前进度总览**：六艺卡片，每艺显示「当前式 / 10」与进度条；可点击进入详情。
- **已完成阶段/最终技**：高亮显示已完成的阶段与已解锁的最终技数量（如 2/6）。
- **快捷入口**：今日打卡、查看统计、分享卡片。

### 4.2 六艺详情页

- 列表展示该艺 1～10 式名称与状态（未开始 / 进行中 / 已完成）。
- 显示每式达标日期（若有）、当前式高亮。
- 入口：开始打卡（跳转打卡页并预选该艺当前式）。

### 4.3 打卡界面

- 日期选择器（默认今天）。
- 按艺添加记录：选择艺 → 选择式 → 组数、次数、是否达标。
- 支持同一天多条记录，保存后写入 `cc_checkins` 并更新 `cc_progress`。

### 4.4 统计页面

- 各艺各阶段耗时（初级/中级/高级/最终技）。
- 最终技完成情况（6 艺中完成数、列表）。
- 连续打卡天数、总打卡天数等（可选）。

## 五、节点4：分享卡片功能

### 5.1 卡片模板设计

- 固定布局：应用名、用户当前进度摘要（六艺当前式或「已达成最终技」）、总耗时/最终技数量、一句励志文案。
- 风格：简洁、可读性强，适合长图分享（如 750×1200 或 1:1.5）。

### 5.2 个人数据提取

- 从 `cc_progress` 与 `cc_checkins` 聚合：当前阶段、每艺耗时、是否完成最终技。
- 不包含敏感信息，仅进度与成就摘要。

### 5.3 分享渠道整合

- **生成图片**：用 Canvas 或 html2canvas 将卡片 DOM 转为图，用户保存相册后自行分享到微信/微博等。
- **Web Share API**：在支持的设备上提供「分享」按钮，分享文本+图片链接或图片文件。

### 5.4 自定义选项

- 可选：昵称/代号、主题色、是否显示「第 X 式」细节或仅显示「初级/中级/高级/最终技」。

## 六、节点5：用户体验优化

### 6.1 初学者引导流程

- 首次进入：简短说明六艺十式与「从第一式开始」的理念。
- 引导完成「选择一艺并做第一次打卡」，完成后标记引导完成，可不再自动弹出。

### 6.2 成就系统

- **徽章**：如「首日打卡」「连续 7 天」「某艺初级完成」「某艺最终技」等。
- **进度条**：每艺 1～10 式进度条；全局「六艺最终技完成数 / 6」。

### 6.3 提醒与激励机制

- 可选浏览器通知（需授权）：每日提醒打卡。
- 应用内：首页展示「已连续 X 天」「距离下一式还差 1 次达标」等文案。

### 6.4 社交互动（可选）

- 不依赖后端时：仅通过「分享卡片」到社交平台实现间接互动。
- 若有后端：可扩展「动态」「点赞」等，列为后续迭代。

## 七、Edges（节点关系）

- **六艺进度系统 → 数据记录系统**：阶段完成标准驱动何时写入达标、何时建议升级。
- **数据记录系统 → 用户界面**：所有展示数据来自本地存储与聚合算法。
- **用户界面 → 分享卡片**：卡片内容从当前进度与成就数据生成。
- **六艺进度系统 → 用户体验优化**：阶段/最终技完成触发成就与庆祝。
- **全节点 → 迭代优化**：通过用户反馈（如设置页「反馈」入口）驱动优先级与改进。

## 九、技术栈建议

- **前端**：React 18 + TypeScript + Vite
- **状态与数据**：React Context + 自定义 hooks，封装 localStorage 读写与进度算法
- **路由**：React Router 6
- **样式**：CSS Modules 或 Tailwind，便于做分享卡片固定布局
- **分享图**：html2canvas 或 Canvas API 绘制卡片并导出 PNG
