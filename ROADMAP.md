# 囚徒健身进度追踪 - 技术路线图与风险

## 一、技术实现路线图

### Phase 1：基础架构与数据层（第 1～2 周）

| 任务 | 产出 |
|------|------|
| 项目初始化 | Vite + React + TypeScript，ESLint/Prettier |
| 六艺十式数据模型 | 常量配置（六艺 ID、名称、十式名称）、TypeScript 类型 |
| 本地存储封装 | `storage.ts`：get/set/remove，键名常量 |
| 进度与打卡类型定义 | `Progress`、`Checkin`、`ArtProgress` 等 interface |
| 进度算法 | 达标判定、当前式推断、阶段耗时计算（纯函数） |

### Phase 2：核心页面（第 3～4 周）

| 任务 | 产出 |
|------|------|
| 路由与布局 | React Router，全局布局（导航、底部 Tab） |
| 首页 | 六艺卡片、当前式/10、已完成阶段与最终技数量、快捷入口 |
| 六艺详情页 | 1～10 式列表、状态、达标日期、跳转打卡 |
| 打卡页 | 日期选择、按艺添加记录（式、组、次、达标）、保存 |
| 统计页 | 各艺阶段耗时、最终技完成列表、连续打卡等 |

### Phase 3：成就与分享（第 5 周）

| 任务 | 产出 |
|------|------|
| 成就系统 | 成就 ID 列表、解锁规则、存储与展示 |
| 庆祝与徽章 | 最终技完成弹窗/动画、成就徽章展示 |
| 分享卡片 | 卡片组件、数据聚合、Canvas/html2canvas 导出 PNG |
| 分享渠道 | 保存图片 + Web Share API（若可用） |

### Phase 4：体验与收尾（第 6 周）

| 任务 | 产出 |
|------|------|
| 初学者引导 | 首次进入引导、首次打卡引导 |
| 设置与反馈 | 提醒开关、反馈链接/邮箱 |
| 响应式与可访问性 | 移动端适配、键盘与焦点、ARIA |
| 构建与部署 | 静态构建、部署到 Vercel/Netlify/OSS） |

---

## 二、用户测试与反馈收集机制

- **应用内**：设置页提供「反馈」入口，链接到 GitHub Issues 或腾讯问卷/金数据。
- **可选**：在关键流程（如完成最终技后）轻量弹窗「觉得好用吗？点这里告诉我们」。
- **数据**：不收集任何可识别用户数据；反馈表单仅收集自愿填写的意见与设备类型（可选）。

---

## 三、上线后迭代优化计划

1. **首月**：根据反馈修 bug、调整文案与引导；必要时增加 1～2 个成就或提醒策略。
2. **第二月**：若存储压力大，引入 IndexedDB 迁移与历史数据导出（JSON 下载）。
3. **后续**：可选功能——多设备同步（需后端）、社交动态、训练计划模板等，按需求优先级排期。

---

## 四、风险评估与应对措施

| 风险 | 等级 | 应对 |
|------|------|------|
| localStorage 被清空 | 中 | 提供「导出数据」为 JSON 文件；引导用户定期导出备份；后续支持导入恢复 |
| 浏览器兼容性（Share API、Notification） | 低 | 特性检测，不支持时隐藏对应按钮或降级为「复制链接/保存图片」 |
| 进度算法与用户预期不一致 | 中 | 提供「手动升级/降级」与「手动标记某式达标」，算法仅作建议 |
| 分享图在某些机型模糊或错位 | 中 | 使用固定尺寸 Canvas、设备像素比处理；多机型真机测试 |
| 用户不知从哪一式开始 | 低 | 引导默认「从第一式开始」；详情页每式简短说明与进阶建议 |
